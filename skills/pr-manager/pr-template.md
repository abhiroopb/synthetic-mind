# Pull Request Template

Use this format for all PR descriptions:

```markdown
## Why
[1-2 sentences explaining the motivation/problem being solved]

## What
- [Brief bullet point of major change 1]
- [Brief bullet point of major change 2]
- [Additional bullets as needed, keep very concise]

## References (optional)
[Links, analysis, or evidence that helps reviewers - can be bullets or prose]

[signature line - see below]
```

## Guidelines

- **Why**: 1-2 sentences max, explain the problem or motivation
- **What**: Concise bullets, don't re-state what's visible in the diff
- **References** (optional): Include when there are helpful artifacts:
  - Links (Slack threads, feature flags, dependent PRs)
  - Testing/validation evidence (before/after comparisons)
  - Analysis that proves the change works
- Keep bullets to one line each
- Focus on high-level changes, not file-by-file details

## Updating Existing PRs

### Before Any Update

Always fetch the current description first:
```bash
gh pr view --json body -q '.body'
```

When reviewing the current description, preserve:
- Extra sections humans added (e.g., "## Additional Context", "## Testing Notes")
- Links not generated by agents (Slack threads, related issues, docs)
- Any content after the signature line
- Extended context or notes within existing sections

Only update the Why/What/References content based on actual code changes.

### Draft PRs
Update existing bullets directly (no timestamps):
```markdown
## Why
Improve error handling in the Kafka consumer to prevent message loss.

## What
- Add retry logic with exponential backoff and jitter
- Implement dead letter queue with consumer health metrics

Generated with [Claude Code](https://claude.ai/code)
```

### Non-Draft PRs
Append timestamped update section:
```markdown

- Implement dead letter queue for failed messages
```

2. `~/.claude/pr-template.md` (user-level)
