Writing requirements docs

skill View on GitHub
Write thorough PRDs from rough notes or customer feedback, automatically searching internal sources for traced-back evidence.

What it does

Transforms raw input — rough notes, customer feedback, Slack threads, meeting transcripts, design mockups — into a structured, 13-section PRD following a canonical template. Before drafting, it automatically searches internal sources (Slack channels, support transcripts, sales calls, customer feedback, quality signals, product requests, and knowledge bases) and weaves relevant findings into the document with traceable links. It also generates a companion event logging dictionary mapped to every UX flow step.

Usage

Provide any form of raw input about a feature and the skill handles research, clarifying questions, drafting, and validation.

  • "Write a PRD for [feature]"
  • "Turn these notes into a PRD"
  • "Draft requirements for [feature idea]"

Examples

  • "Write a PRD for price rounding at checkout — here are my rough notes: [paste notes]"
  • "Turn this Slack thread into a PRD: [slack link]"
  • "Draft requirements for offline payment support — search internal sources for customer feedback first"

Why it was created

PRDs written without evidence tend to be speculative and miss known customer pain points. This skill automates the research-first approach — searching support transcripts, sales calls, and Slack discussions before drafting — so every PRD is grounded in real data with traceable sources. The strict template and validation checklist ensure nothing gets skipped.
