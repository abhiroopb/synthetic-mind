What it is

Not a chatbot, not a pile of prompts. More like a lightweight operating model for AI-assisted PM work.

A morning planner, not just a responder

The first workspace is a Chief of Staff agent that reviews notes, routines, and open workstreams, then writes a launch plan for the day. It decides what should be open before execution begins.

Durable context per thread

Each workstream lives in its own folder with a living CONTEXT.md. The important state is stored on disk, so an interrupted session can resume cleanly without depending on chat history.

Parallel sessions with clear roles

Once the plan exists, the launcher opens the right cmux workspaces in parallel. Planning, project execution, meeting prep, and routine loops can stay separate instead of collapsing into one thread.

A public starter repo, not a black box

The open-source version is sanitized, tool-agnostic, and intentionally lighter than the internal system. It keeps the useful shape while stripping away private integrations and extra machinery.

How a day runs

The whole system starts from one command and a few small files.

1

Launch

start-day.sh validates prerequisites and boots or reuses cmux.

2

Plan

The Chief of Staff workspace reviews notes, routines, and workstreams.

3

Write plan

The planner emits system/today-plan.json with selected workstreams and prompts.

4

Derive state

The public repo mirrors the plan into queue.json and now.json.

5

Resume

Selected workstreams open in parallel and each session picks up from CONTEXT.md.

bash system/start-day.sh
  -> boots or reuses cmux
  -> opens Chief of Staff workspace
  -> Chief of Staff writes system/today-plan.json
  -> build-state-from-plan.py derives queue.json + now.json
  -> launcher opens recommended workstreams
  -> each workstream resumes from its own CONTEXT.md

What changed in the public export

The open-source repo is real, but simplified on purpose.

Lighter state model

The public version uses a simple launch-plan model centered on today-plan.json, then derives the current queue and top recommendation into system/state/.

Generalized automation language

Daemon scripts talk about chat, inbox, docs, and trackers rather than private internal tools, so the patterns are reusable outside my own stack.

Starter workstreams

The repo ships with a small sample set like example-project, checkout-ux, and product-strategy so you can see the shape without inheriting my real work.

Repo-local skills

It includes repo-scoped Amp skills such as running-chief-of-staff and reviewing-product-plans, so the workflow is easier to adapt instead of rebuild from scratch.

The point was portability

I wanted the public repo to show the real operating model without dragging along private names, proprietary integrations, or a bunch of internal infrastructure. The result is a smaller system that still demonstrates the important ideas: planning, continuity, ranking, and resumable parallel work.
